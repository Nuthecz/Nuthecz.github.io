基础知识

chroot

  • change root directory，它用来在指定的根目录下运行程序，简单而言就是将自己指定的目录替换 linux 中默认的 / 根目录，更改进程及其子进程的根目录。

    • chroot("tmp/jail") 将根目录设置为 /tmp/jail ，它创建了一个jail，从而使在这个shell下，我们只能访问这个目录中的内容，形成了一个沙盒

    • 注意：不使用 chdir('/') 的话并不能真正进入 jail，会产生逃逸，这个时候还可以使用 ../../file 来查找 jail 以外的数据

    • # 这时正常用法
chroot("/tmp/jail");
chdir("/");
<!--code￼0-->

    • 同时 linux kernel 不会跟踪进程的先前 chroot，也就是说，我们可以在监狱内再次使用 chroot，然后它会覆盖之前的 chroot

​ 这个程序就是通过 assert(chroot(jail_path) == 0); 来创建 jail，但是不使用 chdir('/'); 就没有作用，所以可以直接逃逸。这里的方法就是关注 sendfile(1, open(argv[1], 0), 0, 128);，它在这个 jail 中打开一个文件，可以推测传输的参数应该需要是真正根目录下的flag。这里分析一下 argv[1] 的不同情况：

  • 如果为 /flag, 那么这个路径会被解析为 /tmp/jail-XXXXXX/flag ，这时假flag的路径
  • 如果为 flag，那么这个结果取决于我们执行脚本的当前工作目录。我们可以在启动流程时设置一个参数 cwd='/' 来逃逸
  • 同时还可以使用路径遍历。如果 argv[1] 为 ../../flag ，也可以进行逃逸。但是使用 ./flag 和 ../flag 也可以成功(在 / 目录下执行 /challenge/babyjail_level1 ../flag)，所以估计这里是因为没有使用chdir，直接执行的当前目录不是 /tmp/jail-XXXXXX，而是真实的当前目录，直接根据当前目录来进行路径遍历即可
  • 还可以使用 openat(AT_FDCWD, "../../flag", 0, 0)，这里AT_FDCWD=0xffffff9c，代表的是current working directory，所以这里就是以当前目录为 .，打开 ../../flag，没有chdir的情况可以进行逃逸

知道什么是openat(int dirfd, const char *pathname, int flags, mode_t mode)，然后这个dirfd必须是目录卡了我好长时间

  • open()函数直接使用传递给它的路径名参数，这意味着它将相对于当前工作目录打开文件。

  • openat()函数接受一个文件描述符参数，以及一个路径名参数。它将路径名解释为相对于由文件描述符指定的目录，而不是当前工作目录。这使得openat()函数更加灵活，并且在多线程或异步I/O环境中更加安全，因为它允许程序员明确地指定打开文件的目录而不必担心在调用之间当前工作目录的改变。此外，openat()也可以用于在chroot环境中安全地打开文件。

level 5

int linkat(int efd,const char *existingpath,int nfd,const char *newpath,int flag);

​ 这里不能用 openat 了，但是可以用 sys_linkat ，这个函数创建一个新目录项 newpath ，它引用现有文件existingpath。 对于linkat函数，现有文件是通过efd和existingpath参数指定的。默认情况下，如果两个路径名中的任意一个是相对路径，那么它需要通过相对于对应的文件描述符进行计算。如果两个文件描述符中的任一个设置为AT_FDCWD，那么相应的路径名（如果它是相对路径）就通过相对于当前目录进行计算。如果任一路径名是绝对路径，相应的文件描述符参数就会被忽略。

socketpair的应用

分析源码

​ 这里主要就是对于 socketpair 的理解。用于创建一对无名的、相互连接的套接字。如果函数成功，则返回0，创建好的套接字分别是sv[0]和sv[1]；否则返回-1，错误码保存于errno中。

  • 这对套接字可以用于全双工通信，每一个套接字既可以读也可以写。例如，可以往sv[0]中写，从sv[1]中读；或者从sv[1]中写，从sv[0]中读；
  • 如果往一个套接字(如sv[0])中写入后，再从该套接字读时会阻塞，只能在另一个套接字中(sv[1])上读成功；

​ 上文使用了下面的语句创建了两个套接字，它们也是一种文件描述符，那么这里parent_socker为3，child_socker为4。

​ 之后就是看父子进程之间的交互了。这里就是首先父进程从标准输入读取我们写的shellcode，然后通过套接字传递给子进程，子进程向父进程发送信息之后执行我们的shellcode。之后父进程在while中读取子进程发送的信息，先打印读取的信息，再截取前9个字节与自己内置的字符串进行比较，如果为 print_msg，就直接输出传递第11个字节后的信息，如果为 read_file，则使用 sendfile 打开子进程发送信息第11个字节开始的数据(这里应该填写一个文件路径)，然后从头开始读128个字节到父端上面。

​ 然后就可以来写汇编了，这里我们结合汇编来分析，汇编总共就三条系统调用。

  • write(4, "read_file:/flag", 0x50)，这里第一轮就是父进程读取shellcode发送给子进程，子进程回个 print_msg，然后父进程输出出来。然后子进程会调用shellcode，这里首先就是向文件描述符4，就是 child_socket，写入了 read_file:/flag，这里就是模拟第一轮的操作，写完之后因为父进程在while循环中不断准备read parent_socket的内容，所以就接收到了字符串 read_file:/flag ，然后解析发现会调用 sendfile 写信息(这里就是打开flag文件了)到 parent_socket。
  • read(4, addr of 0x1337800, 0x50)，有了上面父进程写入flag到 parent_socker，所以按照socket的通信，需要再从子端读出来，这个read的设计就是如此，然后就把flag的内容读取到了地址 0x1337800 的位置了
  • write(4, addr of 0x1337800 + number, 0x50)，读取完之后，这个时候父端和子端都是空闲的了，所以再使用 write 模拟子进程将指定地址的数据写入数据到子端，之后处于while循环的父进程会读取数据，因为这个时候读取的就是flag的内容了，所以前9个字节与内置的字符串都不同，因此这里使用的是上面的printf打印，将传输信息的前9个字节打印出来，所以这里就将flag的9个字节打印出来了，之后再调整写入的地址位置(number)，就可以获得完整的flag了
